需求：Solid.js + Rust 全栈健身 APP，动作库 / 训练计划 / 进度跟踪 / 体测记录 / 视频示范 / 社交。

背景：调研市场发现 Keep 等头部产品重且臃肿，我们想做个轻量版——动作库开源（基于 wger 或 ExRx 数据）、训练计划灵活（自己排或选模板）、进度可视化（PR 记录、容量趋势）、社交可有可无。先跑通核心训练记录 + 计划，社交放在 P2。

核心功能：
- 动作库：每个动作含名称、肌群、设备、难度、视频示范（嵌入 YouTube / B 站）、文字步骤
- 训练计划：自由组合动作成"训练日"（胸日 / 背日 / 腿日），多个训练日组成"周计划"
- 训练记录：实际重量 × 次数 × 组数，组间休息计时（可自定义）
- 进度跟踪：每个动作的 PR（最大重量 / 最多次数），容量趋势（每周总吨数）
- 体测记录：体重、围度（胸 / 腰 / 臀 / 臂 / 腿）、体脂率，趋势图
- 训练日历：日历视图看历史训练，避免连续 3 天练同一个部位
- 视频示范：动作详情页内嵌视频，备网环境降级到本地缓存的缩略图
- 社交（可选）：好友间互相关注、点赞对方 PR 记录、互评（最简版，仅关注流）

涉及模块：
- `backend/src/handlers/`：动作、计划、记录、体测 路由
- `backend/src/db/`：SQLite + sqlx
- `backend/src/services/`：PR 计算、容量聚合、视频元数据
- `frontend/src/routes/`：动作库、训练日历、计划编辑、统计
- `frontend/src/components/`：ExerciseCard、SetLogger、RestTimer、VolumeChart
- `frontend/src/stores/`：Solid signals stores

注意事项：
- 训练记录要按"组"粒度存，不要只存"今日完成 X 公斤"，否则后续分析没法做
- 容量 = Σ(重量 × 次数)，按动作 × 周聚合；空数组除零要保护
- PR 判重要做"是否真正突破"校验：同重量多组不重复计数、助力带完成的不算
- 动作视频用第三方嵌入（YouTube / B 站 iframe），不要自己存视频（带宽贵、版权问题）
- 训练计划切换 / 调整要"冻结旧计划"，新计划从新一周开始，避免历史数据混在一起
- 休息计时用 `setInterval`，但应用切后台要停（iOS / Android 后台限制）
- 训练日历冲突检测：同一肌群 3 天内只允许 1 次训练（避免过度训练）
- 体脂率测量不准，数据点要支持标记"是否早晨空腹"，看趋势更合理
- Solid.js 的 `createResource` 用于服务端数据获取，`createSignal` 本地状态
- Rust 后端异步：用 tokio + axum，DB 池用 sqlx
- 视频嵌入要带 `loading="lazy"`，避免一次加载 20 个视频卡顿
- 动作库初始 200+ 动作，需要 seed 脚本导入（CSV → DB）
- PR / 容量趋势图用 chart.js 或 uPlot（轻量），不要用 recharts（太重）
